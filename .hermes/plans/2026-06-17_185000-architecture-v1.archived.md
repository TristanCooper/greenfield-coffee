## 1. Top-level shape: modular monolith

**Decision:** A single TypeScript codebase deployed as a **modular monolith** at v1, with a clear path to extracting services later if/when needed.

**Why monolith, not microservices:**
- One solo founder cannot operate a microservices platform. The operational cost (deploys, observability, cross-service debugging) dominates everything else.
- The four product modules (M1 Demand, M2 Supply, M3 Money, M4 Compliance) share a single data spine. A microservice split that respects transaction boundaries would still need distributed transactions for cross-module operations (e.g. ship a shipment, which writes to Orders, Stock, and Compliance in one transaction).
- The data model is the moat, not the API. Shipping fast on a single database with strong typing is the highest-velocity path.
- Container overhead for 4 modules is 0; the cost of containers is on the *ops* side, not the *dev* side.

**Why "modular" monolith:**
- The codebase has four distinct bounded contexts. We structure the code so that the contexts are *physically separated* (folders, packages) but *logically* still one deployable.
- The module boundaries are explicit and enforced by code review + linting rules (e.g. "no imports from M2 into M1's domain code"). The boundaries become a forcing function for the v1.5/v2 service extraction if/when it happens.
- Tests can be run per-module. Database migrations can be run per-module. Failure modes are easier to reason about than a true microservice.

**When to revisit:** If any single module becomes a clear bottleneck (compute or team), or if the data residency story demands geographic service separation beyond a single-region database. None of these apply at v1.

## 2. Language and framework

### 2.1 Frontend + API server: Next.js (App Router)

**Decision:** Next.js 15+ (App Router), TypeScript strict mode, React 19, deployed on Vercel.

**Why Next.js for the frontend:**
- The PRD's UX (the daily board, the EUDR opt-out flow, the margin cockpit) is an SPA with rich client interaction. Next.js's React Server Components are useful for SEO-free authenticated apps, but the *client* interactivity is the bulk of the work.
- Vercel is the natural deployment target and handles preview deploys, edge caching, and the operational concerns of a solo founder.
- Server Actions are useful for the *small* set of high-trust mutations (e.g. opt-out sign, DDS sign) that should be POSTed from a server-rendered form, not a client-side fetch.
- The data model and the validation are shared between client and server (Zod schemas), so client-side forms validate against the same rules the server enforces.

**Why Next.js for the API server:**
- Single language (TypeScript) across the whole stack — no separate API service to maintain.
- tRPC (see §2.2) gives us end-to-end types with zero codegen.
- For the PRD's needs, Next.js Route Handlers + Server Actions cover 100% of the API surface. We do *not* need a separate Express/Fastify service.

**When NOT to use Next.js:**
- Long-running connections (WebSockets, server-sent events for live updates). We'll address this in §2.4 (workers + push) — most "live update" needs are actually fine with revalidation.
- CPU-heavy work in the request path. We'll address this in §2.4 (workers).

### 2.2 RPC: tRPC

**Decision:** tRPC for client-to-server communication, v11 with Zod input/output validation.

**Why tRPC:**
- End-to-end types without codegen. The client gets autocomplete on the input shape and the return type from a procedure.
- Zod is the validation library for both client form validation and server input validation. One schema, two consumers.
- Procedures map cleanly onto the PRD's module structure — we get a router per module (`demandRouter`, `supplyRouter`, `moneyRouter`, `complianceRouter`) and a root router that aggregates them.
- tRPC has good Vercel/Next.js integration via the `@trpc/server/adapters/fetch` adapter.

**Tradeoffs:**
- tRPC is not a public API. We expose a thin REST/RPC layer for the v1.5 public API and for webhooks (Shopify, Xero, etc.). The internal API stays tRPC.
- React Query is the client side; we get caching, optimistic updates, and revalidation for free.

### 2.3 ORM: Drizzle

**Decision:** Drizzle ORM, with raw SQL escape hatches for the things Drizzle doesn't model well (PostGIS, hash chain triggers, generated columns).

**Why Drizzle over Prisma:**
- Drizzle's schema is TypeScript-first; Prisma's is a separate DSL. With a solo founder, the schema-in-TS approach is one less thing to context-switch into.
- Drizzle has zero runtime overhead and excellent TypeScript inference. Prisma's query engine is heavier.
- Drizzle has good support for jsonb, generated columns, and Postgres-specific features.
- Drizzle migrations are SQL files generated from the schema diff, with full control over the migration. Prisma's migration story is also good but more opinionated.

**When to drop down to raw SQL:**
- PostGIS queries (geolocation polygon storage, spatial indexes).
- Hash chain triggers (audit log immutability).
- RLS policies.
- Anything where Drizzle's abstractions get in the way of clarity.

### 2.4 Background work: separate worker host

**Decision:** Background work runs on a separate host (Fly.io or Railway), not as Vercel functions.

**Why separate:**
- Vercel functions have a 10-second (hobby) or 60-second (pro) default timeout. Some of our work — net-requirements recompute on a large org, DDS PDF generation with charts, geocoding — can take longer.
- Vercel functions are stateless and ephemeral. Some work needs a long-running process (e.g. the in-scope-check worker for batch shipments).
- The "stuck in a request path" anti-pattern is the most common source of bad UX in SaaS. We want to make the architectural rule explicit: **all work that takes more than 2 seconds goes to the worker**.

**Worker framework:** BullMQ (Redis-backed) is the right default for Node. We use Redis (Upstash or Fly's managed Redis) for the queue. BullMQ has:
- Job retries with exponential backoff
- Cron-style scheduled jobs
- Worker concurrency
- Job priorities
- Dead-letter queues

**Jobs at v1:**
- Net-requirements recompute (event-driven, <5s typical, but can take longer for large orgs)
- Margin recompute (event-driven, fast)
- DDS PDF generation (1-2s typical, but chart rendering can be slow)
- Audit pack PDF generation (similar)
- Audit pack hash chain update (fast, but the *sign* operation is cryptographic and should be outside the request path)
- Daily board warm-cache (scheduled)
- Compliance data completeness check (scheduled)
- Backup verification (scheduled)

### 2.5 Charts and PDFs

**Decision:**
- Charts: Recharts (React-native, simple, no D3 commitment). For the more complex charts (margin waterfall, genealogy graph) we may drop down to D3 later.
- PDF generation: Puppeteer for the audit pack and the DDS (high-fidelity, full HTML/CSS, includes a signature page and a QR code). For the simpler PDFs (packing list, label), use `@react-pdf/renderer` or similar.
- QR code: `qrcode` (server-side) or `qrcode.react` (client-side).
- Map (for the producer geolocation preview): MapLibre GL JS with the OpenStreetMap tile server. We do *not* use Google Maps or Mapbox — both have data-residency and licensing concerns for the UK/EU market. We may swap in a paid tile provider if performance requires it; the abstraction is in place.

### 2.6 What we are NOT using at v1

- **A separate API service (Express/Fastify/standalone).** Next.js Route Handlers cover everything.
- **A frontend framework other than React.** The PRD is React-shaped and the team (you) is React-shaped.
- **A state management library beyond React Query.** Server state via tRPC + React Query; local state via React; form state via React Hook Form. Redux/Zustand/etc. are not justified at v1.
- **A graph database, search engine, or other specialised datastore.** Postgres + jsonb + PostGIS covers the data model. If we need full-text search later, Postgres `tsvector` is the first answer; Algolia/Meilisearch is a v2 conversation.
- **A message broker (RabbitMQ, Kafka).** BullMQ + Redis is enough for v1. Kafka is a v3 conversation if we ever get there.

## 3. Data layer

### 3.1 Database: managed Postgres

**Recommendation:** Postgres 16+ with the **PostGIS extension** enabled, on a managed provider that supports EU and UK regions and has the extension available.

**Providers in priority order:**
1. **Neon** — serverless Postgres, EU region available (Frankfurt), PostGIS support varies (check at signup; their free tier may not include PostGIS but their paid tier does). The branching model is excellent for preview deploys.
2. **Supabase** — managed Postgres, EU region (Frankfurt), PostGIS is officially supported. Comes with auth, storage, and realtime, but we use only the database piece. Reasonable.
3. **Railway** — managed Postgres with EU region (default eu-central-1), PostGIS works. Simpler than Supabase; less mature than Neon for branching.
4. **AWS RDS** — the "serious" option for a regulated workload. PostGIS works. London and Frankfurt regions. More expensive and more ops overhead than the managed-Postgres-tier-1 options. Worth considering at v2 if we need dedicated compute.

**Decision at v1:** Neon. If PostGIS isn't available on the plan we need, fall back to Supabase. (My recommendation: confirm PostGIS availability on Neon's Pro plan before signing up; if not, Supabase is the safer bet.)

**Why Postgres:**
- PostGIS for geolocation polygons (PRD §5.6 `Producer.geolocation`).
- RLS for multi-tenancy isolation on a shared database.
- jsonb for flexible compliance data (the `Producer.certifications`, `Supplier.dds_reference`, `ShipmentEudrDecision.destination_context`, etc.).
- Generated columns for the immutable cost-snapshot patterns (PRD §5.8).
- Strong typing in Drizzle for the whole data model.

### 3.2 Multi-tenancy: schema-per-org or row-level security

**Decision:** **Single database, shared schema, RLS-enforced isolation per org.** The RLS policies are written and applied at v1; not deferred.

**Why RLS, not schema-per-org:**
- Schema-per-org is operationally heavier (per-org migrations, per-org backups, per-org connection pooling). Not viable for a solo founder at v1.
- RLS gives us the same isolation guarantee in the database, with much less operational overhead.
- The PRD already commits to RLS in the data model (PRD §4.3); this decision ratifies it.

**The cost of RLS:** every query needs to be tenant-scoped, which Drizzle doesn't enforce by default. We add a tenant context to the tRPC context, and every procedure calls `withTenantContext(db, orgId, async () => { ... })` which sets the RLS variable. This is friction. The mitigation is *linter-enforced* — every table has an `org_id` column and every query is checked.

**Alternative considered:** application-layer tenant filtering (every query has a `where org_id = $currentOrg`). RLS is more robust because a forgotten filter is a security incident, not a bug. RLS wins.

### 3.3 Caching layer: Redis

**Decision:** Redis (managed, via Upstash or the worker host's managed Redis) for:
- BullMQ job queue
- Session state (if we move off JWT)
- Hot data cache (e.g. the daily board summary, the SKU master)
- Rate limiting (per-IP and per-org)

**Why Redis:** BullMQ requires it; the other use cases are bonus.

**What Redis is NOT for at v1:** It is not the primary database. The product data lives in Postgres. Redis is cache and queue, not source of truth.

### 3.4 Object storage: S3-compatible

**Decision:** S3-compatible object storage (AWS S3 in `eu-west-2` London or `eu-central-1` Frankfurt, depending on the org's residency).

**What's in S3:**
- Audit packs (PDFs and JSON)
- DDS drafts (PDFs and JSON)
- Supplier documents (PDF, KML, GeoJSON)
- Producer geolocation files
- Cupping sheets and other supporting documents
- Invoices generated for accounting
- Signature images (if we add e-signature in v1.5)

**Bucket strategy:**
- One bucket per region (UK and EU).
- Per-org prefix (`s3://coffops-uk/{org_id}/...`).
- All buckets private; access via signed URLs only, with a default 7-day expiry.
- Versioning enabled — accidental overwrites and deletions are recoverable.
- Lifecycle policies: audit pack objects never expire; everything else gets a 7-year default (covers UK FRC 5-year retention + buffer).

## 4. Deployment

### 4.1 Topology

```
                              ┌─────────────────┐
                              │  Vercel         │
                              │  Next.js app    │
                              │  (frontend +    │
                              │   API routes)   │
                              └────────┬────────┘
                                       │
                ┌──────────────────────┼──────────────────────┐
                │                      │                      │
        ┌───────▼────────┐     ┌───────▼────────┐     ┌───────▼────────┐
        │  Neon Postgres │     │  Fly.io /      │     │  S3 (UK or EU) │
        │  (primary DB,  │     │  Railway       │     │  per-org       │
        │   PostGIS,     │     │  (workers,     │     │  prefixes      │
        │   RLS)         │     │   BullMQ)      │     │                │
        └────────────────┘     └───────┬────────┘     └────────────────┘
                                       │
                              ┌────────▼────────┐
                              │  Upstash Redis  │
                              │  (managed)      │
                              └─────────────────┘
```

**Components:**
- **Vercel** — Next.js app, edge CDN, preview deploys per PR, production deploys on merge to main.
- **Neon Postgres** — primary database, with a dev branch per PR (preview deploys read from the branch).
- **Fly.io or Railway** — long-running worker process, running BullMQ workers. Replica count: 1 at v1; scale to 2-3 at v2.
- **Upstash Redis** — managed Redis for BullMQ and caching. (Or use Fly's managed Redis; depends on which worker host we pick.)
- **S3 (AWS London or Frankfurt)** — object storage. Per-org prefix.

### 4.2 Environments

| Environment | Purpose | Data | How it's deployed |
|---|---|---|---|
| Production | Real users | Real orgs, real data | Vercel production, Neon main branch |
| Staging | Pre-production testing | Anonymised copy of production (or a small synthetic dataset) | Vercel preview deploy, Neon staging branch |
| Preview (per PR) | Each PR gets its own environment | Synthetic or anonymised subset | Vercel preview deploy, Neon ephemeral branch |
| Dev (local) | Your laptop | Synthetic data, reset on demand | `pnpm dev`, local Postgres (Docker) |

The Neon branching model is what makes per-PR environments cheap and safe. Every PR creates a branch; the preview deploy reads from it. The branch is destroyed when the PR is closed.

### 4.3 Region strategy

- **Vercel:** runs in the region closest to the user (auto), but the *origin* serverless functions run in a specific region. We pin to `lhr1` (London) for UK orgs and `fra1` (Frankfurt) for EU orgs.
- **Neon:** we provision two databases: one in `eu-west-2` (London) for UK orgs, one in `eu-central-1` (Frankfurt or Dublin) for EU orgs. The application's tenant context routes to the right database.
- **Fly.io / Railway worker:** one worker in each region, talking to the matching database.
- **S3:** one bucket in `eu-west-2` for UK orgs, one in `eu-central-1` for EU orgs.

This is the implementation of the PRD's `Organization.data_residency` field. It's the right architecture for a regulated workload, and it scales — when we add US, we add a US region and a `us-east-1` worker.

### 4.4 Cost (rough v1 estimate)

For a single pilot org with the data volumes implied by the PRD:

- **Vercel Pro:** $20/mo per seat. 1 seat at v1. (The free tier would do for the pilot, but Pro is needed for production.)
- **Neon:** Launch plan is ~$19/mo for the data volumes implied. Includes branching, point-in-time recovery, and PostGIS. Confirm PostGIS on the plan we need.
- **Fly.io (or Railway):** ~$5-15/mo for a single shared-cpu-1x instance with 1GB RAM. Enough for v1.
- **Upstash Redis:** Pay-as-you-go, ~$0.20 per 100k requests. v1 load is trivial; budget $5/mo.
- **S3:** storage + requests. Pilot data is <10 GB; budget $5/mo.
- **Total v1:** ~$55-65/mo.

At 50 paying orgs with 1,000-6,800 kg/mo each, the same architecture holds and adds maybe $50-100/mo (more Postgres compute, more workers, more S3). The first real cost growth is at 200+ orgs; that's a v2 problem.

### 5.2 Module rules

Each module under `/src/server/modules/{demand,supply,money,compliance}` has the same structure:
- `domain/` — pure functions, types, no IO. The business logic.
- `procedures/` — tRPC procedures. The API surface. Imports `domain/`. No direct DB access; uses `db` from `/src/server/db/client.ts`.
- `jobs/` — BullMQ job definitions. Long-running work triggered by domain events.

**Rule:** A module may *call* another module's `domain/` functions, but it may not import another module's `procedures/` or `jobs/`. The import graph is `domain → domain` only, not `domain → procedures`. This is enforced by an ESLint rule.

**What this gives us:** When we extract a module to a service in v2 (say, compliance, which is the most likely candidate), the extraction boundary is already in the code.

### 5.3 Cross-module interactions

Most of the cross-module work happens through the **shared database** and through **events**:

- **Stock movement** (`/src/server/modules/supply/domain/stock-movement.ts`) writes to `StockMovement` and updates lot weights. This is called from:
  - Green receiving (supply procedure → supply domain)
  - Roast event (supply procedure → supply domain)
  - Pack event (supply procedure → supply domain)
  - Sale event (demand procedure → **calls supply domain** for the allocation)
  - Return event (demand procedure → calls supply domain)
- **EUDR decision** (`/src/server/modules/compliance/domain/shipment-eudr-decision.ts`) is called from:
  - Pack & ship flow (supply procedure → calls compliance domain for the in-scope check, the DDS generation, and the opt-out decision)
  - Compliance dashboard (compliance procedure → reads compliance domain)
- **Margin recompute** (`/src/server/modules/money/domain/margin-recompute.ts`) is called from:
  - Any stock movement (BullMQ event)
  - Any sale (BullMQ event)
  - Any landed cost change (BullMQ event)

**Rule:** A procedure may call its own module's `domain/`. A procedure may call another module's `domain/` if the other module exposes a stable API. A procedure may *not* call another module's `procedures/` — that would be a service-to-service call, which is what we're deferring.

## 6. Observability and operations

### 6.1 Logging

**Decision:** Structured JSON logs to stdout; shipped to a log aggregator (Logflare or Axiom) via the Vercel/Fly logging integrations.

**What's logged:**
- Every tRPC procedure call (org_id, user_id, procedure, input_shape, duration, error)
- Every BullMQ job (job_id, type, duration, error)
- Every hash chain event (event_id, type, prev_hash, self_hash)
- Every EUDR decision (decision_id, scope, mode, user, role) — this is the regulatory log
- Every S3 read/write (object_key, action, requester)

**What's NOT logged at v1:** PII beyond `user_id`. We log enough to debug, not enough to be a GDPR incident.

### 6.2 Metrics

**Decision:** Prometheus-compatible metrics exposed by both the Next.js app and the worker. Scrape from a managed Prometheus (Grafana Cloud free tier is the obvious choice).

**What we measure:**
- Request rate, error rate, duration (per procedure)
- Queue depth, job duration, job failure rate (per job type)
- DB query duration (slow query log)
- Daily active orgs, weekly active orgs
- Per-shipment EUDR decision: how many in_scope_dds_filed vs in_scope_opted_out vs out_of_scope
- Cost per pilot org (cloud spend ÷ active orgs)

### 6.3 Alerting

**Decision:** PagerDuty (or similar) for SLO breaches:
- Error rate > 1% over 5 minutes → page
- Queue depth > 1000 jobs → page
- Hash chain validation failure → page immediately (this is the regulator-significant event)
- EUDR decision coverage < 100% on any production org → daily digest to founder (not page; this is a pilot signal not an incident)
- DB connection pool exhaustion → page

At v1, "page" means "page the founder" (you). At v2, "page" means "page the on-call rotation."

### 6.4 Backups and retention

- **Postgres:** Neon's point-in-time recovery gives us 7 days of recovery granularity. For 5-year EUDR retention, we rely on the *primary database* — the data lives there until 5 years + 1 month past creation, then a cron job anonymises PII and keeps the regulatory record.
- **S3:** versioning enabled; lifecycle policy keeps versions for 90 days, then deletes non-current versions. The current version is never deleted automatically.
- **Redis:** ephemeral; no backup. We can rebuild any state from Postgres.
- **Hash chain events:** copied to a write-only S3 bucket daily, as a defense-in-depth measure. If the primary database is ever tampered with, the daily snapshots are the fallback evidence.

## 7. Security and compliance posture

### 7.1 Authentication

- **Email + magic link** at v1 (PRD §7.2). Passwordless, low-friction for solo founders.
- **SSO** is v2.
- **Session storage:** JWT in httpOnly cookie. Short-lived (1 hour); refresh on activity.
- **Rate limiting:** per-IP (10 req/sec) and per-org (50 req/sec) at the edge.

### 7.2 Authorization (RBAC)

- Roles per PRD §5.1: `owner`, `head_roaster`, `packer`, `buyer`, `accountant`, `compliance_officer`, `readonly`.
- Authorization is enforced *in the tRPC procedure*, not the route. A procedure that creates a roast event checks the user's role and the org's RBAC config.
- The opt-out (PRD §6.5) is authorised separately — only `owner`, `compliance_officer`, `head_roaster` can take an opt-out decision, and this is checked at the procedure level.

### 7.3 Data in transit and at rest

- **TLS everywhere.** Vercel, Neon, S3, Fly — all TLS by default.
- **Postgres at rest** is encrypted by the provider (Neon uses AWS KMS).
- **S3 at rest** is encrypted (SSE-S3 or SSE-KMS).
- **Backups at rest** are encrypted by the provider.
- **Per-org data isolation** is enforced by Postgres RLS, as described in §3.2.

### 7.4 GDPR / UK GDPR

- **Data export** endpoint: a roaster can request a full export of their org's data, as JSON, in a single download. Implementation: a worker job that streams all tables for the org, zips, and uploads to S3 with a signed URL.
- **Right to erasure** is more nuanced (regulatory retention applies). We honour erasure for PII (user records, customer records) but not for the regulatory record (5-year retention). The mechanics: pseudonymise the user_id / customer_id on the retained records; keep the regulatory record but remove the personal identifiers.
- **Records of processing activities (RoPA)** is a v1 deliverable. One PDF document, kept current, summarising what data we process and why. (It belongs in the docs, not in code.)
- **Data processing addendum (DPA)** is a v1 legal deliverable, signed by every pilot org. Standard, based on the ICO's template.

### 7.5 EUDR-specific security considerations

- The **hash chain** is the regulatory-grade tamper-evidence layer. Implementation in Session 2.
- The **opt-out audit pack** is the document the roaster hands to a regulator. It must be signed, time-stamped, and reproducible from the hash chain. Implementation in Session 2.
- **Producer geolocation files** (KML, GeoJSON) are stored in S3 with strict per-org access controls. The files are not world-readable; the per-org signed URL is the only access path.
- **DDS submission** (when we add the API in v1.5) requires mTLS to the EU Information System. The certificates are stored in a secrets manager (AWS Secrets Manager or equivalent) and rotated annually.
